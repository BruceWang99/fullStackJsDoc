前端系统设计核心方法论（按常用度排序）

1. 模块化设计（Modular Design）
核心定位：前端最基础、最通用的架构设计方法论，所有前端项目的 “标配”。
核心思想：将代码拆分为高内聚、低耦合的模块，每个模块只负责一个核心功能，模块间通过明确接口通信。
前端落地形式：
JS 层面：ES6 Module、CommonJS 规范（拆分工具函数、业务逻辑）；
CSS 层面：CSS Modules、Scoped CSS、Less/Sass 的嵌套 + 模块化；
目录层面：按功能拆分（如src/utils/、src/components/、src/api/）。
典型目录示例：
plaintext
src/
├── modules/        # 业务模块（如订单、商品）
├── components/     # UI组件模块
├── api/            # 接口请求模块
├── utils/          # 工具函数模块
└── styles/         # 样式模块
适用场景：所有前端项目（从个人小项目到大型企业应用），是其他方法论的基础。
核心价值：解决代码混乱、复用性低的问题，降低单个功能的修改成本。

2. 分层设计（Layered Design）
核心定位：前端中大型项目的核心架构方法论，分离 “关注点” 避免逻辑混杂。
核心思想：将前端代码按 “职责” 拆分为不同层级，层级间单向依赖（下层不依赖上层）。
前端经典分层（从下到上）：
层级	职责	示例目录 / 文件
基础设施层	底层技术能力封装（接口、存储、路由）	src/api/、src/utils/
业务逻辑层	核心业务规则（状态管理、数据处理）	src/store/、src/hooks/
表现层	UI 渲染与用户交互（组件、页面）	src/components/、src/views/
核心原则：表现层不包含复杂业务逻辑，业务逻辑层不依赖 UI 组件，基础设施层屏蔽技术细节。
适用场景：中大型项目（如后台管理系统、电商应用），解决 “组件里塞满业务逻辑” 的问题。

3. 组件驱动开发（CDD, Component-Driven Development）
核心定位：面向 UI 的架构设计方法论，适配 React/Vue/Angular 等组件化框架。
核心思想：先开发独立、可复用的组件，再将组件组装成页面 / 应用，而非从页面反向拆分组件。
核心流程：
拆分组件：原子组件（Button、Input）→ 复合组件（SearchBar、Table）→ 页面组件；
独立开发：用 Storybook 等工具隔离开发组件，脱离业务场景验证 UI / 交互；
测试复用：为组件编写测试，确保复用后行为一致。
适用场景：UI 复杂度高、组件复用需求强的项目（如企业组件库、中台系统）。
核心价值：提升组件复用率，降低 UI 迭代成本，让设计和开发对齐。

4. 领域驱动设计（DDD, Frontend DDD）
核心定位：复杂业务场景的高阶架构方法论，属于 “业务维度的分层 + 模块化”。
核心思想：围绕 “业务领域”（而非技术 / 页面）组织代码，划分限界上下文，让架构贴合业务本质。
前端落地要点：
按业务域拆分（订单域、商品域、用户域），每个域包含自身的模型、API、状态、组件；
域间通过明确接口通信，避免直接耦合；
通用逻辑抽离到 “通用子域”（src/shared/）。
适用场景：超大型项目（多域交互、业务规则复杂），如电商平台、SaaS 系统、大型 ERP。
核心价值：解决 “改一处动全身” 的问题，适配多人协作和长期维护。

5. 原子设计（Atomic Design）
核心定位：UI 架构设计方法论，与 CDD 互补，侧重 “UI 层级拆分”。
核心思想：将 UI 拆分为 5 个层级，从基础到复杂，形成可扩展的 UI 体系：
原子（Atoms）：基础 UI 元素（Button、Icon、Typography）；
分子（Molecules）：原子组合（SearchBar=Input+Button）；
有机体（Organisms）：分子组合（Header=Logo+Nav+SearchBar）；
模板（Templates）：页面骨架（HomeTemplate=Header+Content+Footer）；
页面（Pages）：填充数据的模板。
适用场景：需要统一 UI 规范的项目（如设计系统、企业中台），常与 CDD 结合使用。
核心价值：让 UI 设计和开发标准化，提升界面一致性和复用率。

6. 微前端（Micro-Frontends）
核心定位：大型应用的 “架构拆分” 方法论，解决单应用体积过大、团队协作冲突的问题。
核心思想：将一个大型前端应用拆分为多个独立的 “微应用”（如首页微应用、订单微应用、商品微应用），每个微应用可独立开发、部署、运行。
前端落地方式：
基座应用：负责微应用的加载、路由分发、全局状态共享；
微应用：独立的前端项目（可使用不同技术栈，如 Vue/React/Angular）。
适用场景：超大型企业级应用（如阿里飞书、京东后台），多团队维护同一产品的场景。
核心价值：技术栈解耦、团队解耦、独立部署迭代。
二、不同项目规模的方法论选择建议
项目规模	推荐方法论组合	核心目标
小型项目（<10 页）	模块化设计 + 简易分层（仅区分 UI / 逻辑）	快速开发，降低目录复杂度
中型项目（10-30 页）	模块化设计 + 完整分层设计 + CDD	兼顾效率与可维护性
大型项目（>30 页）	模块化设计 + 分层设计 + Frontend DDD + （可选）微前端	解耦业务，适配多人协作
设计系统 / 组件库	原子设计 + CDD	标准化 UI，提升复用率
总结
基础核心：模块化设计是所有前端项目的基础，分层设计是中大型项目的核心；
UI 聚焦：组件驱动开发（CDD）和原子设计是面向 UI / 组件的核心方法论；
复杂业务聚焦：Frontend DDD 解决业务逻辑耦合问题，微前端解决超大型应用的拆分问题。
这些方法论并非互斥，实际开发中通常组合使用（如 “分层设计 + CDD”“模块化 + DDD”），核心是适配项目复杂度和团队能力，而非生搬硬套。
